# A02:2025 — Security Misconfiguration

## 한 줄 요약

코드는 멀쩡한데 **설정 하나 잘못해서 서버가 통째로 털린다.**

---

## 뭐가 문제야?

| #   | 문제                          | 위험도                                  |
| --- | ----------------------------- | --------------------------------------- |
| 1   | 시크릿 키 하드코딩 (`"1234"`) | 코드 유출 시 세션 전체 위조 가능        |
| 2   | 보안 헤더 없음                | XSS, 클릭재킹 등 브라우저 공격에 무방비 |
| 3   | 에러 메시지에 내부 정보 노출  | DB 종류, 서버 IP 등이 공격자에게 노출   |
| 4   | `debug=True` 상태로 배포      | 브라우저에서 서버 코드 직접 실행 가능   |

---

## 직접 테스트해봐

```bash
# bad.py 실행
python bad.py
```

```bash
# 에러 메시지에 내부 정보 노출 확인
curl http://127.0.0.1:8080/user/999
# → DB 서버 IP, DB 종류까지 그대로 나옴 ❌

# debug=True 인터랙티브 콘솔 확인
# 브라우저에서 http://127.0.0.1:8080/crash 접속
# → 스택트레이스 + 파이썬 콘솔이 브라우저에 노출됨 ❌
```

```bash
# good.py 실행
python good.py
```

```bash
# 에러 메시지 최소화 확인
curl http://127.0.0.1:8081/user/999
# → {"error": "User not found"} 만 반환 ✅

# 보안 헤더 확인
curl -I http://127.0.0.1:8081/user/1
# → X-Frame-Options, CSP 등 헤더 확인 ✅

# 크래시해도 내부 정보 안 나옴
curl http://127.0.0.1:8081/crash
# → {"error": "Internal server error"} 만 반환 ✅
```

---

## 핵심 한 줄

> 설정은 코드가 아니라 **환경변수**로 분리하고, debug 모드는 **절대** 운영 환경에서 켜지 마라.
